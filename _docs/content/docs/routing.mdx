# Routing

Surf uses a radix-tree router. Routes are matched in O(log n); the order they
are registered does not affect performance.

## Basic Routes

```go
app := surf.NewApp()

app.Get("/users", listUsers)
app.Post("/users", createUser)
app.Put("/users/:id", updateUser)
app.Delete("/users/:id", deleteUser)
app.Patch("/users/:id", patchUser)
app.Head("/users", headUsers)
app.Options("/users", optionsUsers)
```

The handler signature is:

```go
type HandlerFunc func(w http.ResponseWriter, r *http.Request) error
```

A returned error is rendered to the client by the [error
renderer](/docs/error-handling) — return `*HTTPError` for a typed response, or
`nil` after writing your own.

## Path Parameters

`:name` captures a path segment:

```go
app.Get("/users/:id", func(w http.ResponseWriter, r *http.Request) error {
    id := surf.Param(r, "id")
    return surf.JSONData(w, map[string]string{"id": id})
})
```

<Callout type="warn">
  Use a single consistent parameter name across the codebase (`:id` everywhere,
  not `:userId` here and `:resourceId` there). Different param names at the
  same tree position silently bind to the first registered name — this is
  inherent to radix routers.
</Callout>

## Wildcard Routes

`*` matches the remainder of the path and is read with `Param(r, "*")`:

```go
app.Get("/files/*", func(w http.ResponseWriter, r *http.Request) error {
    path := surf.Param(r, "*")
    return surf.JSONData(w, map[string]string{"path": path})
})
```

## Per-Route Middleware

Standard `Middleware` (`func(http.Handler) http.Handler`) can be attached to
any route as trailing variadic arguments. It wraps the handler in declaration
order (outermost first):

```go
app.Post("/admin", createAdmin, requireAuth, auditLog)
```

Per-route middleware can short-circuit by not calling `next`, and can
propagate context with `r.WithContext`. It is the recommended way to attach
auth / authz to a single route.

## Route Groups

Groups share a prefix and optionally share middleware:

```go
api := app.Group("/api/v1")

api.Get("/users", listUsers)      // GET /api/v1/users
api.Post("/users", createUser)    // POST /api/v1/users
api.Get("/users/:id", getUser)    // GET /api/v1/users/:id
```

### Group.Use — standard middleware on every route in the group

```go
api := app.Group("/api").Use(requireAuth, surf.RateLimitWithDefaults())
api.Get("/users", listUsers)   // auth + rate limit applied
api.Get("/orders", listOrders) // same
```

### Group.Before / Group.After — legacy `HandlerFunc` wrappers

```go
api := app.Group("/api").Before(func(w http.ResponseWriter, r *http.Request) error {
    w.Header().Set("Content-Type", "application/json")
    return nil
})
```

### Group.Skip — exclude routes from group middleware

A pattern ending in `*` matches by prefix; otherwise it must match the route's
full pattern exactly. **Call `Skip` before registering the affected routes.**

```go
api := app.Group("/api").Before(requireAuth)
api.Skip("/api/health")
api.Get("/health", healthz)   // no auth
api.Get("/users", listUsers)  // auth applied
```

### Nested groups

Nested groups inherit their parent's prefix, before/after handlers, middleware,
and skip patterns:

```go
api := app.Group("/api").Use(requireAuth)

v1 := api.Group("/v1")
v1.Get("/users", listUsersV1)   // /api/v1/users, auth applied

v2 := api.Group("/v2")
v2.Get("/users", listUsersV2)   // /api/v2/users, auth applied
```

## Fast-Path Routes

For hot endpoints, `App.Handle` (and `Group.Handle`) registers a handler that
receives a pooled `*surf.Context` directly. The router copies neither the
request nor allocates per-request state — about twice as fast as the standard
path.

```go
app.Handle("GET", "/users/:id", func(c *surf.Context) error {
    return c.JSONData(map[string]string{"id": c.Param("id")})
})
```

See [Context (Fast Path)](/docs/context) for the full API.

## 404 and 405

When no route matches a path, surf responds with **404 Not Found**. When a
path matches a route registered for a different method, surf responds with
**405 Method Not Allowed** and sets the `Allow` response header listing valid
methods. Both are customizable — see [Error Handling](/docs/error-handling).

## Performance

| Routes | Linear search | Radix tree | Improvement |
|--------|---------------|------------|-------------|
| 10     | ~500 ns       | ~70 ns     | 7×          |
| 100    | ~5.4 µs       | ~74 ns     | 73×         |
| 1000   | ~54 µs        | ~78 ns     | 692×        |
